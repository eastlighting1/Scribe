# 보안 정책 (Security Policy)

Scribe는 아직 초기 단계의 라이브러리입니다. 프로젝트 구조와 주변 스캐폴딩의 상당 부분은 빠른 반복을 통해 먼저 만들어졌고, 이후 테스트, 검증 규칙, 패키징 점검, 워크플로우 점검, 문서 검토를 통해 다듬어지고 있습니다. 따라서 보안 피드백, 정확성 피드백, 그리고 강화 방향에 대한 제안을 초기 단계부터 적극적으로 받고자 합니다.

## 취약점 보고 방법

Scribe에서 보안 문제를 발견했다고 판단되는 경우, 상세 exploit 정보, 비밀값, 토큰, 혹은 비공개 인프라 정보를 공개된 장소에 게시하지 말아 주세요.

이 저장소에서 대부분의 보고는 GitHub 기반 협업 방식을 권장합니다:

- 공개적으로 논의해도 안전한 의심스러운 취약점, 위험한 동작, 검증 누락, 패키징 우려, 의존성 위험, 워크플로우 정책 문제는 GitHub Issue로 열어 주세요.
- 이미 구체적인 보안 강화 수정, 테스트, 워크플로우 개선, 문서 업데이트가 준비되어 있다면 Pull Request로 보내 주세요.
- 아직 공개되면 안 되는 민감한 내용은, 저장소에 해당 기능이 활성화되어 있다면 보안 보고 채널을 사용해 주세요.

Issue, PR, 또는 비공개 보고에는 가능하면 다음 내용을 포함해 주세요:

- 문제에 대한 짧은 설명,
- 영향을 받는 버전 또는 커밋 범위,
- 재현 단계 또는 개념 증명(PoC),
- 예상되는 영향 범위.

어떤 문제가 보안 버그인지 단순 정확성 버그인지 확신이 없더라도, 일단 이슈를 열고 "보안 관련 우려 사항"이라고 명확히 표시해 주시면 됩니다.

## 지원 버전

보안 수정은 우선적으로 가장 최근의 개발 라인을 대상으로 진행됩니다. 별도로 명시되지 않는 한, 이전 버전에 대해서는 수정 사항이 백포트되지 않을 수 있습니다.

## 공개 절차

보고가 접수되면, 유지관리자는 다음 순서에 따라 대응하는 것을 목표로 합니다:

1. 접수 확인,
2. 보고 내용 검증,
3. 수정 또는 완화 조치 준비,
4. 필요할 경우 릴리스 노트에 관련 변경 사항 기록.
